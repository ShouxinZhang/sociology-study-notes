# 面向天赋落地的元属性训练策略

> 基于 `analysis.txt` 中的自述特征定制。
> 核心前提：天赋组合（想象力 + 跨领域联想 + 问题发现 + 抽象结构感 + 自主探索 + 工具借力）已初步识别，当前瓶颈不在天赋本身，而在**元属性**——执行启动力、专注持久力、抗挫折韧性、延迟满足能力、注意力管理。

---

## 一、训练原则：为什么元属性训练必须与天赋使用场景绑定

### 1.1 不要脱离实际场景空练"意志力"

意志力不是一块可以在健身房单独锻炼的肌肉。大量研究（Baumeister 的自我损耗模型已被多次修正）表明，意志力的表现高度依赖**情境**与**动机**。一个人在无聊任务上的"意志力"和在心流任务上的"意志力"完全不同。

对你来说尤其如此：你在看书、看动画、做数学题、编故事时可以自然长时间投入，但在体育运动中很快疲惫。这不是"意志力不够"，而是**你的认知系统对不同任务的能量分配模式不同**。

**核心结论**：不要试图先"练好意志力"再做项目。那是一条死路——因为脱离场景的意志力训练本身就是你最不擅长的任务类型。

### 1.2 元属性训练的真正含义

元属性训练的核心是：

> **在真实的天赋使用过程中，有意识地延伸边界。**

- 不是"我要训练专注力，所以去做一个无聊的任务看自己能坚持多久"
- 而是"我在做一个让我兴奋的原型项目时，有意识地多坚持 15 分钟再休息"

### 1.3 "在做项目的过程中训练意志力"

实际操作原则：

| 错误做法 | 正确做法 |
|---------|---------|
| 先花两周"养成早起习惯"再开始写代码 | 明天就开始写代码，顺便把起床时间早调 15 分钟 |
| 先"学完 React 全部基础"再做项目 | 定一个最小原型，遇到什么学什么 |
| 先"练好专注力"再进入深度工作 | 在深度工作中逐步拉长单次时长 |
| 先"养成记日志的习惯"再追踪进度 | 做完一个 commit 顺手写一行日志 |

**元属性不是前置条件，而是副产品。** 它在你追求真实目标的过程中自然生长。

## 二、执行启动力训练（针对"想法多但启动难"）

### 2.1 为什么启动最难

你的天赋组合（想象力 + 跨领域联想）天然会产生大量"有趣但模糊"的想法。这些想法越多，启动任何一个的阻力就越大——因为大脑在做选择时消耗的能量，和做事本身一样多（决策疲劳）。

同时，你对"完整性"有直觉追求（抽象结构感），所以一旦想到一个项目，脑子里立刻浮现的是最终的完美形态，而不是第一步。**理想状态与起步状态之间的落差，就是启动阻力的主要来源。**

### 2.2 适配版：5 分钟规则

标准版 5 分钟规则是"先做 5 分钟，5 分钟后可以停"。但对你需要做以下调整：

**调整 1：5 分钟规则绑定到具体的启动动作，而不是模糊的"开始做"**

| 不够具体 | 足够具体 |
|---------|---------|
| "写 5 分钟代码" | "打开 VS Code，在 `main.py` 里写出 `def main():` 的函数签名和一行注释" |
| "学 5 分钟数学" | "打开笔记本，把昨天那道题的已知条件重新抄一遍" |
| "做 5 分钟原型" | "在草稿纸上画出系统的三个核心模块的方框图" |

**调整 2：5 分钟后不是"可以停"，而是"重新评估"**

做完 5 分钟后，问自己一个问题：**"我现在想继续吗？"** 如果想继续，不设时限地继续。如果不想继续，允许自己停下来——但记录下这次的启动时间和心理状态。

### 2.3 设计"最小启动单元"

最小启动单元的核心原则：**一个人在最差状态下也能完成的、不需要任何预热的最小动作。**

针对你的三个核心场景：

| 场景 | 最小启动单元 | 完成时间 |
|------|------------|---------|
| 软件开发 | 打开项目 → 读最后一次 commit message → 在代码里写一行 `# TODO:` 注释 | < 2 分钟 |
| 数学/形式化 | 打开笔记 → 用自己的话把当前问题重新写一遍（不需要解题） | < 3 分钟 |
| 写作/内容创作 | 打开文档 → 在末尾写一句"今天我想到的一个想法是……" | < 1 分钟 |

**关键：最小启动单元的目标不是产出，而是让你的身体和注意力进入"工作姿态"。**

### 2.4 快速反馈循环设计

你明确提到喜欢"可以及时获得正反馈"的任务。利用这个特点：

**策略 1：缩小反馈单元**

不要把反馈锚定在"项目完成"上，而是锚定在每一个微小进展上：

- **写代码**：每写完一个函数，立刻运行测试 → 绿灯亮 = 正反馈
- **学数学**：每理解一个定理，用自己的话写一句总结 → 写出来 = 正反馈
- **做原型**：每实现一个小功能，截图放到进度文档里 → 看到变化 = 正反馈

**策略 2：可视化进度**

建一个极简的进度跟踪，推荐方式：

```
# progress.md

## 本周进度
- [x] 2026-04-06: 完成了 main.py 的基础框架
- [x] 2026-04-07: 实现了数据读取模块
- [ ] 2026-04-08: 实现输出格式化
```

每完成一项勾一个 `[x]`，这个动作本身就是正反馈。

**策略 3：设置"今日最小胜利"**

每天早上（或前一晚），定义一个**必然能完成**的最小胜利：

- "今天至少写 10 行代码"
- "今天至少读 2 页论文并写 1 句笔记"
- "今天至少画 1 个系统模块图"

完成后在日志里标记 ✓。连续标记 7 天以上，正反馈的累积效应会显著增强启动力。

### 2.5 每日/每周执行模板

**每日启动模板**（建议放在固定位置，每天花 2 分钟填写）：

```
日期：____
今日最小胜利：____（一句话，必须是今天必然能完成的）
启动动作：____（最小启动单元是什么）
实际开始时间：____
实际结束时间：____
完成情况：✓ / ✗
一句话复盘：____
```

**每周回顾模板**（每周日花 10 分钟）：

```
本周完成的最小胜利数：____ / 7
启动失败的次数：____
启动失败的主要原因：____
下周需要调整的启动策略：____
本周最满意的一次执行：____
```

## 三、专注持久力训练（针对"容易疲惫"和"深度工作时长不足"）

### 3.1 番茄钟进阶用法：渐进式深度工作

标准番茄钟（25 分钟工作 + 5 分钟休息）对你来说可能有两个问题：
1. 如果你正在心流状态，25 分钟打断反而有害
2. 如果任务本身不够吸引你，25 分钟可能太长

**适配方案：弹性番茄钟 + 渐进拉长**

**阶段 1（第 1-2 周）：找到你的基线**

先用几天测量你当前在不同任务上的自然专注时长：
- 数学/编程/写作：设个计时器但不设闹钟，记录你自然中断的时间
- 这个时间就是你的"当前基线"

**阶段 2（第 3-4 周）：基线 + 5 分钟**

在基线时长后，有意识地多坚持 5 分钟：
- 如果你的数学基线是 20 分钟，目标调到 25 分钟
- 如果你的编程基线是 35 分钟，目标调到 40 分钟
- 不需要每次都成功，每周有 4 次以上成功即可

**阶段 3（第 5-8 周）：每两周 + 5 分钟**

持续缓慢拉长，直到达到 45-60 分钟。对于你的认知类型，60 分钟的单次深度工作已经足够高效——不需要追求 3 小时不间断。

**关键：不要跟别人比时长。你的天赋在于思维密度而非思维时长。**

### 3.2 生理层面：认知续航的硬基础

你提到"打乒乓球羽毛球一会就困倦"，这可能暗示你的神经系统在高强度反应任务上消耗较快。以下建议直接影响认知续航：

**睡眠**（影响力最大，但最容易被忽略）

| 项目 | 建议 |
|------|------|
| 总时长 | 7-8 小时（非 6 小时也非 9 小时，找到你的甜点） |
| 固定时间 | 起床时间比就寝时间更重要——固定起床时间 ± 30 分钟以内 |
| 睡前 1 小时 | 不看手机屏幕上的短视频/社交媒体（看书/看动画可以） |
| 午睡 | 15-20 分钟的午睡对认知恢复非常有效，但不要超过 30 分钟 |

**运动**（选择对你有效的类型）

你不擅长高反应性运动（乒乓球、羽毛球），但跑步和跳远中等偏上。建议：
- **有氧运动**：每周 3 次，每次 20-30 分钟慢跑或快走。有氧运动对前额叶功能（执行控制、专注力）有直接提升效果
- **不必强迫自己做不擅长的运动**：运动的目的是维持认知续航，不是成为运动员

**饮食**（简要原则）

- 避免高碳水午餐导致的下午昏沉（"饭困"）：午餐适当减少精制碳水，增加蛋白质和蔬菜
- 咖啡因策略：如果喝咖啡/茶，在上午 10 点到下午 2 点之间使用，避免下午 4 点以后摄入
- 保持饮水：认知功能对脱水非常敏感，桌上放一杯水

### 3.3 区分"真正的认知疲劳"和"假疲劳"

这是一个被严重低估的问题。你提到在体育运动中容易困倦，但在看书、看动画、编故事中可以长时间投入。这说明你的"疲劳"很多时候不是真的认知资源耗尽，而是**任务与认知偏好不匹配导致的动力衰减**。

**快速判断方法**：

当你感到"累了、不想继续了"时，做以下测试：

1. **切换测试**：换一个你感兴趣的认知任务（比如从刷题切换到编一个小故事）。如果 3 分钟内你恢复了精力 → 刚才是假疲劳（任务不匹配）
2. **身体测试**：站起来走动 2 分钟，喝杯水。如果回来后精力恢复 → 刚才是久坐导致的生理疲劳
3. **如果以上两种都不管用** → 可能是真正的认知疲劳，需要休息 15-30 分钟

**针对假疲劳的策略**：

- 重新审视当前任务：是不是进入了"打磨细节"而不是"探索新结构"的阶段？你的天赋更喜欢后者
- 把当前任务重新框架为"问题"：不是"我要继续写这段无聊的代码"，而是"这个模块有没有更优雅的结构？"——这会重新激活你的问题发现能力

### 3.4 深度工作渐进计划

| 阶段 | 时间 | 单次深度工作目标 | 每日深度工作总量 | 策略 |
|------|------|----------------|----------------|------|
| 基线期 | 第 1 周 | 自然时长（测量） | 不强制 | 只记录，不干预 |
| 起步期 | 第 2-3 周 | 基线 + 5 min | 1-2 个深度块 | 弹性番茄钟 |
| 增长期 | 第 4-6 周 | 30-40 min | 2-3 个深度块 | 加入"假疲劳检测" |
| 稳定期 | 第 7-8 周 | 40-50 min | 3 个深度块 | 开始交错不同任务类型 |
| 目标期 | 第 9 周+ | 45-60 min | 3-4 个深度块 | 维持，不追求更多 |

> 注意：每日 3-4 个 45-60 分钟的深度工作块（总计 3-4 小时真正的深度工作），已经超过了绝大多数知识工作者的实际水平。不要追求 8 小时深度工作——那不现实也不必要。

## 四、抗挫折与情绪恢复训练（针对"遇到难题或学不会时容易放弃"）

### 4.1 成长型心态的具体实践（不是喊口号）

"成长型心态"这个概念被用滥了，但它的核心操作其实很朴素：

**操作 1：重新标注困难**

当你遇到"学不会"的感觉时，在心里把标签从：
- ❌ "我不行" / "我没天赋"
- 改成 → ✅ "我还没找到正确的理解路径"

这不是自我欺骗。你的学习特征是"通过理解关系和建立联系来记住"，而不是"通过重复来记住"。当你"学不会"时，往往不是能力问题，而是当前的学习材料没有让你建立起连接。

**操作 2：当日困难记录**

每次遇到卡住的时候，立刻花 30 秒写下：

```
卡住了。
- 在做什么：____
- 卡在哪里：____
- 我试过什么：____
```

写下来这个动作本身就有减压效果。而且这些记录日后会变成极有价值的学习路径文档。

**操作 3：降低粒度而不是放弃**

"学不会"很多时候是因为粒度太大。把"学会 X"拆成更小的单元：

- "学会机器学习" → "理解线性回归的损失函数在做什么"
- "学会 Lean4" → "运行一个 `hello world` 级别的证明"
- "学会范畴论" → "用自己的话解释什么是态射"

### 4.2 "卡住协议"：系统化的脱困步骤

当你在学习或开发中卡住时，按以下顺序执行（不要跳步）：

```
卡住协议 v1.0

第 1 步：暂停 2 分钟（不是继续死磕）
         → 站起来，喝口水，做 3 次深呼吸

第 2 步：写下"我到底卡在哪里"（30 秒）
         → 一句话描述。模糊的"学不会"不算，要具体到
           "我不理解 X 和 Y 之间的关系"

第 3 步：切换理解路径（5 分钟）
         → 换一个来源重新理解同一个概念：
           - 如果刚才在看文字 → 去找一个视频/图示
           - 如果刚才在看教科书 → 去搜 Stack Overflow / Reddit
           - 如果刚才在单独思考 → 去问 LLM 一个具体问题

第 4 步：降低难度（如果第 3 步无效）
         → 当前问题太难？那这个问题的"前置问题"是什么？
           退回到前置问题，先理解前置问题

第 5 步：标记并跳过（如果以上都无效）
         → 在笔记里标记"暂时搁置"，附上你目前的理解程度
         → 切换到另一个任务
         → 24 小时后（睡一觉后）再回来看——
           很多时候隔天再看就通了（离线学习效应）
```

### 4.3 "失败数据库"：把卡住变成素材

建立一个极简的失败/卡住记录：

```markdown
# stuck_log.md

## 2026-04-06
- **任务**：理解 Monad 的概念
- **卡在**：不理解 bind 操作和普通函数组合的区别
- **已试**：看了 Haskell wiki，看了一个 YouTube 视频
- **状态**：暂时搁置
- **后续**：（等理解了再来填）

## 2026-04-08
- **回到上面那条**：看了一个用"盒子"比喻的解释后理解了——
  Monad 不是一种函数，而是一种"在带上下文的值之间串联计算"的抽象模式
- **关键洞察**：我需要先有一个具体例子（Maybe Monad）才能理解抽象定义
- **学到的学习策略**：遇到抽象概念时，先找最小的具体例子
```

这个记录的价值在于：
1. **把挫败感转化为数据**——你不是在失败，你在收集理解路径的数据
2. **帮你识别自己的学习模式**——时间一长你会发现，你每次"学会"的突破口都有规律
3. **提供给问题发现能力新的素材**——你的天赋之一是发现联系，而失败数据库就是联系的原始素材

### 4.4 情绪日志（每天 2 分钟）

不需要长篇大论。每天结束时花 2 分钟填写：

```
日期：____
今天的能量等级（1-5）：____
最消耗能量的事情：____
最恢复能量的事情：____
一句话情绪总结：____（例如："上午高效，下午卡住后有点沮丧，晚上看了个有趣的项目恢复了"）
```

坚持 2 周后你会看到明确的模式：什么消耗你，什么恢复你。这比任何理论都有用。

## 五、延迟满足与长期坚持训练

### 5.1 把长期目标拆成可见的里程碑

你的一年天赋验证目标是"软件开发能力"。这个目标太大了，大脑无法直接追踪进度。需要拆成**有明确完成标准的里程碑**：

**里程碑拆解示例**：

```
🏔 终极目标：一年后验证软件开发能力是否属于我的天赋组合
    ├── Q1：完成 1 个可运行的个人项目原型
    │     ├── M1：选定项目主题 + 技术栈
    │     ├── M2：核心功能 MVP 可运行
    │     └── M3：有 README + 可在 GitHub 展示
    ├── Q2：完成 1 个有用户反馈的产出
    │     ├── M4：在某个社区/平台发布
    │     ├── M5：收到至少 3 条有效反馈
    │     └── M6：根据反馈迭代一次
    ├── Q3：完成 1 个更复杂的系统原型
    │     └── ...
    └── Q4：回顾与判断
          └── ...
```

**关键**：每个里程碑必须有**二元判断标准**（完成 / 未完成），不能是"差不多了"。

### 5.2 "承诺设备"：用外部机制辅助坚持

你的自主探索能力很强，但"自主"也意味着缺乏外部压力时容易随兴趣漂移。以下是三种适合你的外部承诺机制：

**机制 1：公开承诺（社交压力）**

- 在 GitHub 上每周 push 至少一次代码
- 考虑在社交平台上定期分享进度（不需要长文，一句话 + 截图即可）
- 如果有信任的朋友或社群，告诉他们你的季度目标

**机制 2：自动化提醒**

- 设置每日固定时间的提醒（比如下午 2 点的"今天写代码了吗"提醒）
- 用 GitHub 的 contribution graph 作为视觉追踪——连续绿色方块的压力比任何 app 都有效
- 考虑设置每周日晚上的周回顾提醒

**机制 3：同伴问责**

找一个也在做自己项目的人（不需要做同一个方向），约定：
- 每周互相发一句话进度更新
- 不评判对方的进度，只确认"这周做了 / 没做"
- 这种轻量的社交问责对长期坚持效果很好

### 5.3 把长期坚持变成"探索游戏"

你已经有很强的自主探索能力，问题只是如何让长期项目保持"探索"的感觉，而不变成"强制苦行"。

**策略 1：给项目设置"支线任务"**

长期项目不可能每天都是兴奋的核心探索。总有需要打磨细节的"平淡期"。在平淡期插入"支线任务"来维持动力：

- 主线：完成原型的数据库模块（必须但不太有趣）
- 支线：花 30 分钟探索一个你好奇的新技术/新概念，写一段读书笔记

支线任务的作用是**给大脑中的联想模块提供新输入**，让你在回到主线时可能产生新的灵感。

**策略 2：进度"解锁"机制**

像游戏一样设计解锁条件：

| 解锁条件 | 奖励 |
|---------|------|
| 连续 7 天完成最小胜利 | 花半天时间纯粹做发散探索（不需要有产出） |
| 完成一个里程碑 | 买一本你想看的书 / 看一部想看的动画 |
| 坚持 30 天记日志 | 回顾日志，写一篇"我学到了什么"的总结 |

**策略 3：允许"战略性走神"**

严格禁止一切"走神"是反人性的，对你这种联想型思维尤其如此。改为：
- 每天分配一个固定的 15-20 分钟"自由联想时间"，在这个时间里可以追任何兔子洞
- 但在深度工作时间里，如果冒出了新想法，不要立刻追，而是花 10 秒写在旁边的纸上，做完当前任务再回来看

## 六、注意力与信息输入管理

### 6.1 控制发散联想不变成无限追兔子洞

你的跨领域联想能力是天赋，但如果不加管理，它也会变成最大的效率杀手。核心问题是 **exploration vs. exploitation** 的平衡。

**"想法捕捉 → 延迟处理"系统**：

1. 随身携带一个极简的"想法收集器"（一个 txt 文件或纸质便签）
2. 工作过程中冒出新想法时，用 10 秒写下关键词，**不要展开**
3. 每天结束时花 5 分钟回顾这些想法，决定：
   - 🗑️ 丢弃（大多数想法冷却后会变得不那么吸引人）
   - 📥 归档（有价值但不紧急，放入"想法仓库"）
   - ⭐ 行动（确实值得投入时间，排入下周计划）

这个系统的关键在于：**你没有压制想法，只是延迟了处理时间。**

### 6.2 信息输入的"预算制"

每天的清醒时间大约 16 小时。建议的分配预算：

| 类型 | 建议时长 | 说明 |
|------|---------|------|
| 专注执行 | 3-4 小时 | 写代码、解题、写文章等有明确产出的活动 |
| 定向学习 | 1-2 小时 | 为当前项目服务的学习（看文档、看教程、读论文） |
| 发散探索 | 0.5-1 小时 | 自由浏览、追兔子洞、阅读不相关的有趣内容 |
| 日常事务 | 2-3 小时 | 吃饭、通勤、家务、社交等 |
| 休息/娱乐 | 3-4 小时 | 看动画、看书、玩游戏、聊天等 |
| 睡眠 | 7-8 小时 | 不可压缩 |

**重点规则**：
- 发散探索有上限（每天不超过 1 小时），但不是零——零会压制你的天赋
- 专注执行和定向学习必须在发散探索之前完成（先吃饭再吃甜点）
- 如果某天发散探索产生了一个极好的想法——记录下来，但仍然在第二天再决定是否行动

### 6.3 社交媒体和娱乐内容管理

极端方案（"删除所有社交媒体 app"）对你不适用，因为你的联想素材很多来自这些渠道。务实策略：

**策略 1：时间盒子而非完全禁止**

- 设定每天最多 2 次检查社交媒体，每次不超过 15 分钟
- 用手机的"屏幕使用时间"功能设置每日 app 限额
- 深度工作时间段内，把手机放在另一个房间或开勿扰模式

**策略 2：区分"输入型娱乐"和"消耗型娱乐"**

| 输入型（对你有养分） | 消耗型（纯消耗注意力） |
|-------------------|-------------------|
| 看有故事深度的动画/电影 | 无目的地刷短视频 |
| 读激发思考的小说/文章 | 反复刷社交媒体信息流 |
| 和朋友深入讨论一个问题 | 在评论区里和陌生人争论 |
| 玩有策略深度的游戏 | 机械性地刷成就/日常任务 |

不需要完全消灭"消耗型娱乐"（你是人不是机器），但要意识到它的存在，并控制其占比。

**策略 3：睡前保护**

- 睡前 1 小时避免社交媒体——不是因为"蓝光对眼睛不好"（这个影响被夸大了），而是因为社交媒体的信息流会激活你的联想系统，导致入睡困难
- 用阅读或轻松的内容替代

## 七、可执行的 30 天启动计划

### 第 1-7 天：基线自测 + 环境准备

**目标**：了解自己当前的真实水平，准备好基础工具。

| 日 | 行动清单 |
|----|---------|
| D1 | □ 设定固定起床时间（明天开始执行）<br>□ 准备一个"想法捕捉器"（txt 文件或便签）<br>□ 创建 `progress.md` 进度文档<br>□ 创建 `stuck_log.md` 卡住日志<br>□ 创建每日情绪日志模板 |
| D2 | □ 测量深度工作基线：选一个你感兴趣的任务，不设闹钟，记录自然中断时间<br>□ 填写第一条情绪日志 |
| D3 | □ 再测量一次深度工作基线（用不同类型的任务）<br>□ 开始记录"想法捕捉"——工作中冒出的新想法写关键词 |
| D4 | □ 计算两天的平均深度工作基线时长<br>□ 选定接下来要做的项目/学习方向<br>□ 为该项目定义第一个里程碑 |
| D5 | □ 设置每日提醒（固定时间的"今天做了吗"提醒）<br>□ 设置手机屏幕使用时间限额<br>□ 整理工作环境（物理桌面 + 电脑桌面） |
| D6 | □ 写出项目的最小启动单元（具体到一个动作）<br>□ 回顾这 5 天的情绪日志，初步观察模式 |
| D7 | □ **第一次周回顾**<br>□ 回答：基线时长是多少？最难启动的时刻是什么？<br>□ 下周目标：每天完成最小胜利 + 深度工作基线 + 5 分钟 |

### 第 8-21 天：核心习惯建立期

**目标**：建立"最小胜利 + 渐进深度工作 + 每日记录"的基本循环。

**每日固定动作**：

```
□ 起床 → 固定时间 ± 30 分钟
□ 每日启动模板填写（2 分钟）
□ 完成最小启动单元
□ 至少 1 个深度工作块（基线 + 5 分钟）
□ 想法捕捉（全天，每次 10 秒）
□ 想法回顾（每天结束时，5 分钟）
□ 情绪日志（每天结束时，2 分钟）
□ 标记今日最小胜利是否完成：✓ / ✗
```

**每周额外动作**（D14, D21）：

```
□ 周回顾（10 分钟）
□ 本周最小胜利完成率：____ / 7
□ 深度工作时长变化趋势
□ 卡住日志是否有积累
□ 下周是否需要调整策略
□ 如果连续 3 天以上没完成最小胜利 → 反思：是最小胜利设得太大了，
  还是有其他外部因素？
```

**第 8-14 天重点**：
- 巩固最小启动单元 → 让"每天至少做一点"变成不需要思考的默认动作
- 深度工作时长目标：基线 + 5 分钟

**第 15-21 天重点**：
- 引入"卡住协议"——遇到困难时按步骤执行，而不是凭感觉
- 深度工作时长再增加 5 分钟
- 尝试一天做 2 个深度工作块（如果精力允许）

### 第 22-30 天：调整与固化

**目标**：根据前三周的数据调整策略，让系统变得自然。

| 日 | 行动清单 |
|----|---------|
| D22 | □ 回顾前 21 天所有情绪日志和卡住日志<br>□ 识别 3 个模式（什么让我高效？什么让我低效？什么让我恢复？） |
| D23-D27 | □ 继续每日固定动作<br>□ 根据 D22 的发现调整：如果下午总是低效 → 把深度工作移到上午<br>□ 如果某类任务总是卡住 → 调整学习路径或降低粒度 |
| D28 | □ **第四次周回顾**<br>□ 全月数据汇总：最小胜利完成率、深度工作时长变化、卡住次数 |
| D29 | □ 写一份简短的"30 天复盘"（500 字以内）：<br>　1. 我建立了哪些习惯？<br>　2. 哪些策略有效？<br>　3. 哪些策略需要扔掉或修改？<br>　4. 下一个 30 天的重点是什么？ |
| D30 | □ 决定下一个 30 天的计划<br>□ 庆祝——你完成了第一个 30 天周期 |

### 每周回顾问题清单

每周日花 10 分钟回答以下问题：

```
1. 本周完成了几天最小胜利？（____ / 7）
2. 本周深度工作平均单次时长是多少？比上周变化了多少？
3. 本周卡住了几次？用了卡住协议吗？有效吗？
4. 本周最消耗我能量的是什么？
5. 本周最恢复我能量的是什么？
6. 下周我要调整什么？（只选 1 件事）
7. 用一句话总结本周：____
```

## 附录：核心理念速查

| 理念 | 一句话 |
|------|--------|
| 元属性是副产品 | 不要先练意志力再做事，而是在做事中练意志力 |
| 最小启动单元 | 降低每次开始的阻力到"最差状态也能做"的程度 |
| 弹性番茄钟 | 先测基线，再每两周 + 5 分钟，不跟别人比时长 |
| 假疲劳检测 | 切换任务 / 站起来走动 / 两者都不管用才是真疲劳 |
| 卡住协议 | 暂停 → 写下卡在哪 → 换路径 → 降难度 → 搁置并跳过 |
| 失败数据库 | 把每次卡住变成未来的学习素材 |
| 想法延迟处理 | 10 秒记关键词，当天晚上再决定是否行动 |
| 发散有预算 | 每天 0.5-1 小时发散探索，不是零也不是无限制 |
| 承诺设备 | 公开承诺 + 自动化提醒 + 同伴问责，三选至少一 |
| 进度可视化 | 用 checklist、contribution graph、里程碑让进度"看得见" |
